Web安全-跨站脚本攻击（XSS）

概念

通常指通过「HTML注入」篡改了网页，插入了恶意脚本，从而在用户浏览网页时，控制用户浏览器的一种攻击

针对各种不同场景产生的 XSS，需要区分情景对待

成因

XSS 漏洞成因是由于动态网页的 Web 应用对用户提交请求参数未做充分的检查过滤，允许用户在提交的数据中掺入 HTML 代码（最主要的是「>」、「<」），然后未加编码地输出到第三方用户的浏览器，这些攻击者恶意提交代码会被受害用户的浏览器解释执行。

类别

  • 反射型 XSS，只是简单的把用户输入的数据「反射」给浏览器。也叫做「非持久型 XSS 」
  • 存储型 XSS，把用户输入的数据「存储」在服务器端。也叫做「持久型 XSS 」
  • DOM Based XSS，通过修改页面的DOM节点形成的XSS

防御

HttpOnly

浏览器禁止页面的 JavaScript 访问带有 HttpOnly 属性的 Cookie

主要解决 XSS 后的 Cookie 劫持攻击

可以有选择性的加在任何一个 Cookie 值上

输入检查

同时在客户端和服务器端代码实现输入检查

一般是检查用户输入的数据中是否包含一些特殊字符，如果发现特殊字符则将这些字符过滤或者编码

输入的数据可能会被展示在多个地方，每个地方的语境可能各不相同，如果使用单一的替换操作，则可能会出现问题

输出检查

HtmlEncode、JavascriptEncode、XMLEncode、JSONEncode等编码方式，合适的场景使用合适的编码方式

输出位置 防御方法
在HTML标签中输出 HtmlEncode
在HTML属性中输出 HtmlEncode或HTMLEntities
在script标签中输出 JavascriptEncode
在事件中输出 JavascriptEncode
在CSS中输出 encodeFor
在地址中输出 检测http协议之后再进行URLEncode

富文本处理

使用白名单方式，只允许比较安全的标签存在

原则

  • 不要在允许位置插入不可信数据
  • 在向HTML元素内容插入不可信数据前对HTML解码
  • 在向HTML常见属性插入不可信数据前进行属性解码
  • 在向HTML JavaScript Data Values插入不可信数据前，进行JavaScript解码
  • 在向HTML 样式属性值插入不可信数据前，进行CSS解码
  • 在向HTML URL属性插入不可信数据前，进行URL解码
